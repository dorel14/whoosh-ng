# Implementation Plan for Release 1.0.0

## Overview
This plan outlines the remaining tasks to prepare for the 1.0.0 release of the whoosh-NG project. Tasks are ordered logically and should be completed sequentially where dependencies exist.

---

## Tasks

### 1. Version Bump
- **Action**: Update version from `1.0.0.dev0` to `1.0.0` in:
  - `pyproject.toml` (or equivalent version file)
  - `__init__.py` or version module if applicable
- **Command**: 
  ```bash
  # Example using poetry (adjust if using other tools) => on utilise uv
  poetry version 1.0.0
  # Or manual edit of version files
  ```

### 2. Commit Documentation Scaffolding (Requires User Approval)
- **Files to commit**:
  - `docs/en/` (directory)
  - `docs/fr/` (directory)
  - `docs/guides/` (directory)
  - `docs/_config.yml`
  - `docs/Gemfile`
  - `docs/make.bat`
  - `docs/Makefile`
- **Actions**:
  1. Verify all documentation scaffolding files are present and correct
  2. Stage the files: `git add docs/en docs/fr docs/guides docs/_config.yml docs/Gemfile docs/make.bat docs/Makefile`
  3. **Present the staged changes to the user for approval**
  4. If approved: `git commit -m "docs: add initial documentation scaffolding"`
  5. If not approved: Address user feedback and repeat

### 3. Commit Finalized Documentation Files
- **Files to commit**:
  - All finalized English documentation in `docs/en/`
  - All finalized French documentation in `docs/fr/`
  - Any updated configuration files in `docs/` if modified during finalization
- **Actions**:
  1. Verify all documentation content is finalized and approved
  2. Stage the files: `git add docs/en/* docs/fr/*` (adjust paths as needed)
  3. **Present the staged changes to the user for approval**
  4. If approved: `git commit -m "docs: add finalized English and French documentation"`
  5. If not approved: Address feedback and repeat

### 4. Run Complete Test Suite on Multiple Python Versions
- **Action**: Execute the full test suite across supported Python versions
- **Commands** (adjust based on project setup):
  ```bash
  # Using tox (if configured)
  tox
  
  # Or using tox with specific environments
  tox -e py38,py39,py310,py311
  
  # Or using nox
  nox
  
  # Or using pytest with tox-like setup
  pytest --cov=whoosh tests/
  ```
- **Verification**: All tests must pass across all specified Python versions

### 5. Verify Documentation Builds Without Warnings
- **Actions**:
  1. Navigate to docs directory: `cd docs`
  2. Install documentation dependencies: `bundle install` (if using Jekyll) or follow project docs
  3. Build documentation: `make html` or `bundle exec jekyll build` or `make html` (depending on setup)
  4. Check for warnings/errors in the build output
  5. Fix any warnings before proceeding
  6. Return to root: `cd ..`

### 6. Check Package Installation in Clean Environment
- **Actions**:
  1. Create a fresh virtual environment: `python -m venv test-env`
  2. Activate it: `source test-env/bin/activate` (Linux/Mac) or `test-env\Scripts\activate` (Windows)
  3. Install the package: `pip install .` or `pip install .[dev]` if extras needed
  4. Verify installation: `pip show whoosh-ng` (or package name)
  5. Deactivate: `deactivate`
  6. Clean up: `rm -rf test-env`

### 7. Confirm Entry Points Work
- **Actions**:
  1. Activate a fresh test environment (or reuse from step 6)
  2. Install the package: `pip install .`
  3. Test each entry point defined in `pyproject.toml`/`setup.py`:
     - Example: `whoosh --help` (if CLI exists)
     - Example: `python -c "import whoosh; print(whoosh.__version__)"`
  4. Verify all entry points execute without error
  5. Deactivate and clean up test environment

### 8. Create Release Tag and Generate GitHub Release
- **Actions**:
  1. Ensure all changes are committed and pushed to main
  2. Create annotated tag: `git tag -a v1.0.0 -m "Release 1.0.0"`
  3. Push tag: `git push origin v1.0.0`
  4. Generate GitHub Release:
     - Via GitHub UI: Draft new release, tag v1.0.0, add release notes
     - Or via GitHub CLI: `gh release create v1.0.0 --title "Release 1.0.0" --notes "Release notes here"`
  5. Ensure GitHub Actions CI runs successfully for the release

### 9. Clean Up Pre-existing Ruff-Format Debt
- **Actions**:
  1. Check current Ruff status: `ruff check .`
  2. Fix auto-fixable issues: `ruff check --fix .`
  3. Format code: `ruff format .`
  4. Verify no remaining issues: `ruff check .` (should show no errors)
  5. Commit changes: `git add . && git commit -m "style: fix ruff-formatting issues"`

---

## Execution Notes
- **User Approval Required**: Steps 2 and 3 require explicit user approval before committing
- **Sequential Dependencies**: 
  - Steps 1-3 should be completed before running tests (step 4)
  - Steps 4-7 should pass before creating release (step 8)
  - Step 9 (ruff cleanup) can be done early but should be verified before final commit
- **Error Handling**: If any step fails, address the issue before proceeding
- **Final Verification**: After step 8, verify the release is correctly published on GitHub

---

## Estimated Time
- Version bump: 5 minutes
- Documentation commits: 10 minutes (plus user review time)
- Test suite: 5-30 minutes (depends on CI setup and test duration)
- Documentation build: 2-5 minutes
- Installation test: 3-5 minutes
- Entry point verification: 2-3 minutes
- Release tagging: 5 minutes
- Ruff cleanup: 5-15 minutes

**Total active time**: Approximately 30-60 minutes (excluding user review time and test execution time)
